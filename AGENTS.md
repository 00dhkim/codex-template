작업을 시작하기 전에 요구사항이 모호하거나 확인이 필요한 사항이 있으면, 진행 전에 사용자에게 질문한다.

## 공통 작업 원칙

### 사용자 판단이 필요한 것은 먼저 묻는다

- 기본 원칙은 `판단이 필요한 선택은 사용자에게 먼저 묻기`다.
- 예외는 사용자가 `개입 없이 끝까지 진행하라`고 명시한 경우다.
- 이 예외 상황에서는 Codex가 사용자를 대신해 임시 판단을 내려 진행할 수 있다.

### 새 프로젝트를 만들면 full access 스킬 사용 여부를 먼저 제안한다

- 사용자가 새 프로젝트, 새 리포지토리, 새 worktree, 초기 scaffold 생성을 요청하면 초기에 `$project-full-access` 스킬 사용 여부를 짧게 제안한다.
- 이 스킬은 현재 프로젝트의 `.codex/config.toml`에 `sandbox_mode = "danger-full-access"`와 `approval_policy = "never"`를 설정하는 용도다.
- 사용자가 원할 때만 적용하고, 명시적 요청 없이 자동 적용하지 않는다.

### Codex 요청은 `목표 + 제약 + 검증 + 완료조건`으로 받는 쪽이 낫다

- 목표가 불분명하면 먼저 결과물의 형태를 확인한다.
- 제약이 없으면 공개 API, 데이터 형식, 외부 인터페이스를 함부로 바꾸지 않는다.
- 검증 명령이 없으면 수정 범위를 기준으로 최소 검증을 스스로 고른다.
- 완료조건이 없으면 사용자가 무엇을 끝으로 볼지 짧게 확인한다.

### Codex는 탐색, 수정, 검증 순서로 움직인다

- 먼저 관련 파일, 실행 경로, 테스트, 기존 패턴을 찾는다.
- 여러 파일을 건드리면 수정 전에 어떤 파일을 왜 바꾸는지와 어떤 검증을 돌릴지 짧게 공유한다.
- 수정 후에는 가능한 한 실제 검증을 실행한다.
- 검증을 못 돌렸다면 이유와 사용자가 이어서 실행할 정확한 명령을 남긴다.

### 요청이 넓으면 단계로 쪼개 해석한다

- 한 번에 너무 많은 축을 바꾸는 요청이면 `영향 범위 조사 -> 최소 수정 -> 후속 정리` 순서로 나눈다.
- 사용자가 우선순위를 안 줬으면 회귀 위험이 큰 축부터 다룬다.

### 보고 형식도 요구사항의 일부로 본다

- 사용자가 원하는 최종 답변 형식이 있으면 그대로 따른다.
- 형식 지시가 없으면 변경 내용, 검증 결과, 남은 리스크를 우선 요약한다.
- 코드 리뷰 요청이면 버그, 리스크, 테스트 누락을 먼저 보고한다.

## 공통 품질 원칙

### 의미 있는 단위로 커밋하고 기존 변경을 덮어쓰지 않는다

- 여러 개념을 한 커밋에 무리하게 섞지 않는다.
- 커밋 메시지는 `제목 1줄`로 끝내지 말고, 가능하면 `본문`도 함께 적는다.
- 제목은 변경의 핵심을 한 줄로 요약하고, 본문에는 `무엇을 바꿨는지`, `왜 바꿨는지`, `검증은 어떻게 했는지`를 짧게 남긴다.
- 문서, 테스트, 스크립트, 버그 수정처럼 성격이 다른 변경이 함께 들어가면, 본문에서 항목을 나눠 설명한다.
- 사용자가 이미 수정한 내용이나 다른 작업자의 변경은 먼저 읽고 충돌 범위를 파악한다.
- 명시적 요청 없이 기존 변경을 되돌리거나 파괴적인 명령을 쓰지 않는다.

### 컨텍스트보다 기존 패턴을 먼저 확인한다

- 저장소에 이미 있는 스크립트, 테스트, 실행 경로, 문서 패턴을 먼저 읽는다.
- 새 흐름을 만들기 전에 기존 엔트리포인트를 재사용할 수 있는지 본다.
- 프로젝트 맥락이 바뀌면 같은 구현 습관을 그대로 복제하지 않는다.

### 정확도나 동작 의미에 영향 주는 전처리와 인터페이스 변경은 검증한다

- 리사이즈, padding, chunking, normalization, schema 변경처럼 결과 의미를 바꿀 수 있는 수정은 검증 없이 밀어넣지 않는다.
- 손실 가능성이 있는 축소나 변환은 샘플, 테스트, 지표 중 최소 하나로 확인한다.

### 기본 검증은 약한 쪽이 아니라 충분한 쪽을 고른다

- 검증 지시가 없으면 수정 범위를 기준으로 최소한의 안전한 검증을 스스로 선택한다.
- 여러 계층을 동시에 건드렸거나 판단이 애매하면 더 강한 검증을 고른다.
